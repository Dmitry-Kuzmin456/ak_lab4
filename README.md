Проект реализует ассемблер, транслятор в бинарный машинный код, микропрограммную
модель процессора и набор golden tests для проверки инструментальной цепочки.

## Язык программирования

Язык программы - ассемблер. Программа состоит из директив, меток, инструкций и
данных. Транслятор выполняет простую двухпроходную сборку: сначала вычисляет
адреса меток и размещение секций, затем генерирует бинарный код и debug dump.

### Синтаксис

Комментарии начинаются с `;` и продолжаются до конца строки. Символ `#`
используется для immediate-операндов.

BNF-подобное описание:

```bnf
<program>      ::= { <line> }
<line>         ::= [ <label> ":" ] [ <statement> ] [ <comment> ]
<label>        ::= <identifier>
<statement>    ::= <directive> | <instruction> | <data-item>

<directive>    ::= ".section" ("text" | "data")
                 | ".org" <number>
                 | ".const" <identifier> <number>
                 | ".macro" <identifier> { <identifier> } <macro-body> ".endmacro"
                 | ".ifdef" <identifier>
                 | ".ifndef" <identifier>
                 | ".ifconst" <identifier>
                 | ".else"
                 | ".endif"

<instruction>  ::= <mnemonic>
                 | <mnemonic> <operand-list>
<operand-list> ::= <operand> { "," <operand> }
<operand>      ::= <data-register>
                 | <address-register>
                 | <special-register>
                 | "(" <address-register> ")"
                 | "#" (<number> | <char> | <identifier>)
                 | <number>
                 | <char>
                 | <identifier>

<data-register>    ::= "R1" | "R2" | "R3"
<address-register> ::= "A1" | "A2" | "A3"
<special-register> ::= "ZERO"
<data-item>        ::= ".word" (<number> | <char> | <identifier>)
                     | ".cstr" <string>

<number>       ::= decimal | hexadecimal-with-0x-prefix
<char>         ::= "'" <one-character> "'"
<string>       ::= "\"" { <character> } "\""
<identifier>   ::= letter { letter | digit | "_" }
```


### Семантика языка

Стратегия вычислений последовательная: инструкции выполняются в порядке роста
адресов памяти команд, пока поток управления не изменён командой перехода или не
выполнена остановка.

Все метки имеют глобальную область видимости. Метка в секции `text` обозначает
адрес байта в памяти команд, метка в секции `data` обозначает адрес слова в
памяти данных. Точка входа задаётся обязательной меткой `_start` в секции
`text`. Если `_start` отсутствует или объявлена не в `text`, транслятор
завершает сборку ошибкой.

Типизация минимальная и машинная:

- инструкция хранится в памяти команд как opcode и типизированные операнды;
- слово данных хранится в памяти данных как 32-битное знаковое значение;
- символы представлены кодами Unicode/ASCII, которые затем выводятся младшим
  байтом;
- `.cstr` размещает каждый символ в отдельном 32-битном слове памяти данных и
  добавляет нулевой терминатор;
- immediate-операнды пишутся как `#value` и занимают 2 байта signed payload;
- прямые адреса данных и адреса переходов занимают 2 байта payload;
- косвенная адресация пишется как `(A1)`, `(A2)` или `(A3)`.

Макросы раскрываются текстово до разбора секций и меток. Поддерживаются
позиционные параметры:

```asm
.macro PRINT label
    OUT_CSTR label
.endmacro

PRINT msg
```

Условная компиляция работает по объявленным константам и макросам:

- `.ifdef NAME` --активен, если `NAME` определён как `.const` или `.macro`;
- `.ifndef NAME` - активен, если `NAME` не определён;
- `.ifconst NAME` - активен, если `NAME` определён;
- `.else` и `.endif` завершают условный блок.

## Организация памяти

Архитектура памяти - Harvard:

- память команд: `2^16` ячеек по 8 бит;
- память данных: `2^16` ячеек по 32 бита;
- память микрокода: адресное пространство до 256 микрокоманд, занятые
  микрокоманды упакованы подряд.

Память команд и память данных физически разделены в модели. Инструкции не
смешиваются со статическими данными и строками.

```text
Registers
+------------------------------------+
| R1 R2 R3 A1 A2 A3 IP CR PS ZERO   |
+------------------------------------+

Command memory, 8-bit cells
+--------------------------------+
| 0000 : opcode                   |
| 0001 : operand kind / count     |
| 0002 : operand payload / kind   |
| 0003 : ...                      |
| ...                             |
+--------------------------------+

Data memory, 32-bit cells
+--------------------------------+
| 0000 : .word / .cstr char       |
| 0001 : .word / .cstr char       |
| ...                             |
+--------------------------------+
```

Регистры процессора:

| Регистр | Размер | Назначение |
| --- | ---: | --- |
| `R1..R3` | 32 бита | Регистры данных общего назначения. |
| `A1..A3` | 16 бит | Адресные регистры для косвенной адресации. |
| `ZERO` | 32 бита | Специальный регистр, чтение всегда возвращает `0`, запись игнорируется. |
| `IP` | 16 бит | Instruction pointer, адрес следующего байта команды. |
| `CR` | 8 бит | Command register, opcode текущей инструкции. |
| `PS` | 4 бита | Флаги `N`, `Z`, `V`, `C`. |

Отображение сущностей языка на память:

- инструкции секции `.section text` размещаются в памяти команд;
- статические данные секции `.section data` размещаются в памяти данных;
- `.org` задаёт следующий адрес размещения в текущей секции;
- `.word` занимает одно 32-битное слово данных;
- `.cstr` занимает `len(string) + 1` слов данных;
- регистры данных используются как явные операнды команд;
- адресные регистры могут хранить адрес ячейки памяти данных;
- `label` как операнд обозначает прямое обращение к памяти данных;
- `#label` обозначает непосредственное значение адреса метки.

## Система команд

Архитектура команд - регистровая CISC. Мнемоники единые, а режим адресации
задаётся синтаксисом операнда:

- `R1`, `R2`, `R3` - регистр данных;
- `A1`, `A2`, `A3` - адресный регистр;
- `ZERO` - специальный регистр, всегда равный нулю;
- `label` - прямое обращение к памяти данных;
- `(A1)` - косвенное обращение к памяти данных по адресу из `A1`;
- `#value` - immediate.

Реализованы сложные инструкции:

- `OUT_CSTR addr` одной ISA-командой выводит null-terminated строку из памяти
  данных через порт `0`;
- `POLY x, c0, c1, ..., cn, dst` вычисляет
  `c0 + c1*x + ... + cn*x^n` и имеет переменное число операндов.

### Кодирование

Байт opcode всегда первый. Для обычных инструкций количество операндов
определяется opcode. Только инструкции с переменным числом операндов, сейчас это
`POLY`, хранят вторым байтом фактическое количество операндов. После этого
каждый операнд кодируется как тег типа и фиксированный 2-байтный payload:

| Тип | Код | Payload |
| --- | ---: | --- |
| `R1..R3` | `0x01` | 2 байта номера регистра |
| `A1..A3` | `0x02` | 2 байта номера регистра |
| `label` | `0x03` | 2 байта адреса данных |
| `(A1)..(A3)` | `0x04` | 2 байта номера адресного регистра |
| `#value` | `0x05` | 2 байта signed int16 |
| `port` | `0x06` | 2 байта номера порта |
| `label` перехода | `0x07` | 2 байта адреса команды |
| адрес `.cstr` | `0x08` | 2 байта адреса данных |
| `ZERO` | `0x09` | 2 байта номера спецрегистра |

| Инструкция | Opcode | Семантика |
| --- | ---: | --- |
| `HLT` | `0x01` | Остановить модель. |
| `NOP` | `0x02` | Нет операции. |
| `MOV src, dst` | `0x03` | `dst <- src`, без изменения флагов. |
| `ADD src, dst` | `0x04` | `dst <- dst + src`. |
| `SUB src, dst` | `0x05` | `dst <- dst - src`. |
| `INC dst` | `0x06` | `dst <- dst + 1`. |
| `DEC dst` | `0x07` | `dst <- dst - 1`. |
| `CMP src, dst` | `0x08` | Выставить флаги для `dst - src`. |
| `BEQ addr` | `0x09` | `IP <- addr`, если `Z = 1`. |
| `BNE addr` | `0x0A` | `IP <- addr`, если `Z = 0`. |
| `BLT addr` | `0x0B` | `IP <- addr`, если `N = 1`. |
| `BGT addr` | `0x0C` | `IP <- addr`, если `N = 0` и `Z = 0`. |
| `JMP addr` | `0x0D` | `IP <- addr`. |
| `IN port, dst` | `0x0E` | `dst <- input_port[port]`. |
| `OUT src, port` | `0x0F` | `output_port[port] <- src & 0xFF`. |
| `OUT_CSTR addr` | `0x10` | Вывести C-строку из памяти данных через порт `0`. |
| `MUL src, dst` | `0x11` | `dst <- dst * src`. |
| `DIV src, dst` | `0x12` | `dst <- dst / src`, целочисленно. |
| `MOD src, dst` | `0x13` | `dst <- dst % src`. |
| `POLY x, c0, ..., cn, dst` | `0x14` | `dst <- c0 + c1*x + ... + cn*x^n`. |

`JUMP addr` поддерживается как псевдоним `JMP addr`.

### Такты и микротакты

Модель исполняется с точностью до такта. Каждый tick выполняет одну
микрокоманду.

Fetch-декодирование выполняется микрокомандой `FETCH`: из памяти команд
читается opcode, затем operand descriptors. Для обычных инструкций их количество
берется из opcode, а для `POLY` дополнительно читается байт `operand_count`.
После этого `IP` сдвигается на длину инструкции и `uPC` переходит к
микропрограмме opcode.

Ориентировочные длительности:

| Класс инструкции | Полный цикл |
| --- | ---: |
| `HLT`, `NOP`, переходы, `IN`, `OUT`, `OUT_CSTR` | 2 такта |
| `MOV` | 5 тактов |
| `INC`, `DEC` | 5 тактов |
| `CMP` | 6 тактов |
| `ADD`, `SUB`, `MUL`, `DIV`, `MOD` | 8 тактов |
| `POLY` | `3 + K` тактов, где `K` -- количество коэффициентов `c0..cn` |

## Транслятор

Транслятор запускается как консольная программа:

```bash
uv run python -m src.translator program.asm program.bin
```

Аргументы:

- `program.asm` -- входной текст на ассемблере;
- `program.bin` -- выходной бинарный файл;

Транслятор всегда создаёт человекочитаемый debug dump рядом с бинарным файлом:
для `program.bin` будет создан или перезаписан `program.bin.log`.

Этапы трансляции:

1. Очистка строк от комментариев и пустых строк.
2. Препроцессинг макросов и условной компиляции.
3. Разбор `.section`, `.org`, `.const`, меток, инструкций и данных.
4. Первый проход: вычисление адресов в `text` и `data`.
5. Второй проход: генерация памяти команд, памяти данных и бинарного файла.
6. Генерация debug dump рядом с бинарным файлом.

Формат debug dump:

```text
0000 - 03030004010000 - MOV max_i, R1
0007 - 0301000003000C - MOV R1, i
0023 - 08030003010000 - CMP min_factor, R1
```

## Формат бинарного файла

Машинный код хранится в настоящем бинарном формате `AK4B`, а не в текстовом
виде.

```text
offset  size  description
0       4     magic bytes: "AK4B"
4       2     entry point, uint16 big-endian
6       2     command memory image length, uint16 big-endian
8       2     data memory image length, uint16 big-endian
10      N     command memory image, bytes
10+N    4*M   data memory image, int32 words, big-endian
```

Память команд сохраняется плотным префиксом до последнего использованного байта.
Память данных также сохраняется плотным префиксом до последнего использованного
слова. Если `.org` создаёт пропуск, в бинарный образ попадают нулевые слова.

## Модель процессора

Модель процессора запускается как консольная программа:

```bash
uv run python -m src.machine program.bin --input input.txt --trace trace.log
```

Вывод в файл:

```bash
uv run python -m src.machine program.bin --input input.txt --output output.txt
```

Аргументы:

- `program.bin` - бинарник формата `AK4B`;
- `--input input.txt` - файл входного потока;
- `--output output.txt` - файл для результата моделирования;
- `--trace trace.log` - путь к журналу тактов;
- `--limit N` - максимальное число тактов, по умолчанию `1000000`;

## Запуск asm одной командой

Для запуска можно сразу собрать и исполнить исходный asm-файл:

```bash
uv run run_code program.asm --input input.txt --output output.txt --trace trace.log
```

Аргументы:

- `program.asm` - входной текст на ассемблере;
- `--input input.txt` - необязательный файл входного потока;
- `--output output.txt` - необязательный файл для результата моделирования;
- `--trace trace.log` - необязательный путь к журналу тактов;
- `--limit N` - максимальное число тактов, по умолчанию `1000000`;

Команда всегда сохраняет бинарный файл и debug dump рядом с исходником: для
`program.asm` будут созданы или перезаписаны `program.bin` и `program.bin.log`.
Если `--trace` не указан, журнал пишется рядом с исходником в `program.trace`.
Если `--output` не указан, результат программы печатается в терминал.

### DataPath

DataPath содержит:

- память команд;
- память данных;
- регистры данных `R1`, `R2`, `R3`;
- адресные регистры `A1`, `A2`, `A3`;
- служебные регистры `IP`, `CR`, `PS`;
- АЛУ с операциями `ADD`, `SUB`, `MUL`, `DIV`, `MOD`, `INC`, `DEC`;
- входной и выходной stream buffers.

Основной поток данных:

1. `FETCH` декодирует opcode и типизированные операнды.
2. `SELECT_SRC` или `SELECT_DST` выбирает operand descriptor из instruction
   info register.
3. `READ_SELECTED_TO_SRC` и `READ_SELECTED_TO_DST` читают значения из регистров,
   памяти, immediate или косвенно через адресный регистр.
4. `ALU` выполняет операцию и выставляет флаги.
5. `WRITE_SRC_TO_SELECTED` или `WRITE_ALU_TO_SELECTED` записывает результат в
   выбранный операнд.
6. Для портов и `OUT_CSTR` используются специальные микропереходы ControlUnit.

### ControlUnit и микрокод

![Схема Control Unit](docs/scheme_cu.png)


ControlUnit микропрограммный. Микрокод хранится в плотно упакованном массиве
`microcode_memory`. Каждая ячейка содержит микрокоманду:

| Поле | Назначение |
| --- | --- |
| `op` | Микродействие: `FETCH`, `SELECT_SRC`, `SELECT_DST`, `READ_SELECTED_TO_SRC`, `READ_SELECTED_TO_DST`, `ALU`, `WRITE_SRC_TO_SELECTED`, `WRITE_ALU_TO_SELECTED`. |
| `alu` | Операция АЛУ для микродействия `ALU`. |
| `move` | Правило изменения `uPC`: `NEXT`, `FETCH`, `DISPATCH_OP`, `HLT`, `BRANCH`, `IN`, `OUT`, `CSTR_LOOP_OR_FETCH`, `POLY_START`, `POLY_STEP`. |

Основные адреса микропрограмм:

| `uPC` | Назначение |
| ---: | --- |
| `0` | `FETCH` и dispatch по opcode. |
| `1..2` | `HLT`, `NOP`. |
| `3..6` | `MOV`. |
| `7..41` | `ADD`, `SUB`, `MUL`, `DIV`, `MOD`. |
| `42..54` | `INC`, `DEC`, `CMP`. |
| `55..59` | Ветвления и `JMP`. |
| `60..61` | `IN`, `OUT`. |
| `62` | `OUT_CSTR`. |
| `63..64` | `POLY`. |

Ветвления и port I/O используют микрокоманду `FETCH`, а конкретное решение
принимается в ControlUnit:

- для `BEQ/BNE/BLT/BGT` проверяются флаги `PS`;
- для `IN` проверяется номер порта и наличие данных во входном буфере;
- для `OUT` младший байт указанного источника добавляется в выходной буфер;
- для `OUT_CSTR` используется специальный переход `CSTR_LOOP_OR_FETCH`, который
  либо завершает инструкцию на `0`, либо выводит символ и возвращается к чтению
  следующего слова строки.
- для `POLY` используются микропереходы `POLY_START` и `POLY_STEP`, которые
  циклом проходят по переменному списку коэффициентов.

### Журнал тактов

Trace пишется построчно и потоково, чтобы длинные программы не держали весь
журнал в памяти. Пример строки:

```text
tick=61 mode=scalar uPC=043 IP=0027 CR=05 R1=999 R2=0 R3=0 A1=000C A2=0000 A3=0000 PS=N0Z0V0C0
```

Строка журнала содержит:

- номер такта;
- режим `scalar`;
- `uPC`;
- `IP`, `CR`;
- `R1..R3`, `A1..A3`;
- флаги `N`, `Z`, `V`, `C`;
- событие ввода-вывода, если оно произошло.

## Ввод-вывод

Вариант `stream` реализован через буферы:

- при старте модель читает весь файл `--input` в список кодов символов;
- `IN 0, R1` извлекает один символ из входного буфера и кладёт его в `R1`;
- если входной буфер пуст, моделирование останавливается;
- `OUT R1, 0` добавляет `R1 & 0xFF` в выходной буфер;
- после завершения модель возвращает весь выходной буфер как строку.

Вариант `port` реализован специальными инструкциями `IN port, dst` и
`OUT src, port`.
Поддерживается порт `0`. Обращение к другому порту считается ошибкой модели.
`OUT_CSTR` также пишет в порт `0`, но номер порта в операнде не передаётся:
операндом является адрес C-строки.

## Строки

Вариант строк -- `cstr`, то есть null-terminated строки. Статические строки
задаются директивой `.cstr` в секции данных:

```asm
.section data
msg:
    .cstr "OK"
```

В памяти данных это размещается так:

```text
DataMem[msg + 0] = 'O'
DataMem[msg + 1] = 'K'
DataMem[msg + 2] = 0
```

Один символ занимает одно 32-битное слово памяти данных. Это неэффективно, но
так было удобнее реализовать в этой модели.

Строки можно выводить двумя способами:

- обычным циклом на asm через `MOV`, `CMP`, `OUT`, `INC`;
- CISC-командой `OUT_CSTR addr`, которая выполняет такой цикл микрокодом.

## Алгоритм варианта: `prob1`

Вариант `prob1` -- Project Euler problem 4: найти наибольший палиндром,
являющийся произведением двух трёхзначных чисел.

Программа находится в [prob1.asm](prob1.asm), golden-версия --
[tests/golden/prob1.yml](tests/golden/prob1.yml).

Алгоритм:

1. `i` идёт от `999` до `100`.
2. `j` идёт от `990` до `100` с шагом `11`.
3. Шаг `11` используется потому, что каждый шестизначный палиндром делится на
   `11`; значит, хотя бы один множитель должен быть кратен `11`.
4. Считается `product = i * j`.
5. Если `product <= best`, внутренний цикл можно завершить для текущего `i`.
6. Иначе число проверяется на палиндром:
   - `candidate` делится на `10`;
   - остаток через `MOD 10` добавляется к `reversed`;
   - `reversed = reversed * 10 + digit`;
   - цикл идёт до `candidate == 0`.
7. Если `reversed == product`, обновляется `best`.
8. После завершения перебора `best` печатается по цифрам через `DIV`/`MOD`.

Результат:

```text
906609
```

У алгоритма нет внешних параметров. Поэтому файл входного потока для этого теста
пустой, а результат всё равно передаётся через стандартный output stream
процессора.

## Тестирование

Golden tests запускаются командой:

```bash
uv run pytest
```

Golden tests организованы в формате `tests/golden/<case>.yml`, аналогично
примеру `CSA-lab4`: один YAML-файл содержит исходный asm, входной поток, лимит
тактов и ожидаемые результаты. Для каждого кейса тест:

1. транслирует `in_source` в бинарный файл `AK4B`;
2. сравнивает полный debug dump `out_code_log`;
3. запускает модель процессора с `in_input`;
4. сравнивает полный вывод `out_stdout`;
5. проверяет устойчивые фрагменты trace из `out_trace_contains`.

Полный trace больших программ не хранится в эталоне, чтобы не держать в
репозитории сотни тысяч строк. Вместо этого проверяются репрезентативные
события: ввод-вывод, ключевые состояния регистров и отдельные такты выполнения.

Покрытые golden-сценарии:

## Пример полного запуска

Трансляция:

```bash
uv run python -m src.translator prob1.asm prob1.bin
```

Debug dump будет записан в `prob1.bin.log`.

Запуск модели:

```bash
uv run python -m src.machine prob1.bin --trace trace.log
```

Ожидаемый вывод:

```text
906609
```
